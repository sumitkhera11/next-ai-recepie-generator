🚀 Example: Using It For AI Recipe Generation

Imagine you have server action:

export async function generateAIRecipe(prompt) {
  const response = await fetch("/api/generate", {
    method: "POST",
    body: JSON.stringify({ prompt }),
  });

  if (!response.ok) {
    throw new Error("Failed to generate recipe");
  }

  return response.json();
}


Now in your Client Component:

"use client"

import useFetch from "@/hooks/use-fetch"
import { generateAIRecipe } from "@/actions/recipes"

export default function AIRecipeForm() {
  const { data, loading, fn: generate } = useFetch(generateAIRecipe)

  const handleSubmit = async () => {
    await generate("Make a healthy chicken pasta")
  }

  return (
    <div>
      <button onClick={handleSubmit}>
        {loading ? "Generating..." : "Generate Recipe"}
      </button>

      {data && <pre>{JSON.stringify(data, null, 2)}</pre>}
    </div>
  )
}

💡 Why This Is Better Than Direct Fetch

Without hook, every component would need:

const [loading, setLoading] = useState(false)
const [error, setError] = useState(null)


And try/catch every time.

Now?

Just:

const { data, loading, fn } = useFetch(...)


Much cleaner.

🧠 Why This Pattern Is Smart

Your hook:

✔ Centralizes error handling
✔ Automatically shows toast
✔ Reusable for ANY async action
✔ Keeps UI clean
✔ Scales well

This is production-level pattern.

⚠️ One Small Improvement Suggestion

Right now:

const [loading, setLoading] = useState(null);


Better to initialize as:

const [loading, setLoading] = useState(false);


Because loading should be boolean.

BOOM.

💡 Why Use ...args ?

Because your hook becomes reusable.

Today:

getMealsByCategory(category)


Tomorrow:

generateAIRecipe(prompt, userId)


Then:

args = ["Make pasta", 123]


And this works:

await cb("Make pasta", 123)


Hook doesn’t care how many parameters.

It just forwards them.

That’s the power.
